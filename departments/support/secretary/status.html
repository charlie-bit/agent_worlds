0. 这是什么（TL;DR）

secretary 是 agent_worlds support 部门里、唯一在 critical path 之外的角色， 也是第一个对外部世界开放的角色。它每天扫一遍我的各个 inbox（Slack / Jira / GitHub…）， 把"落了什么、什么需要我"整理成一份带日期的 digest，让事情不再 slip。

本会话定调： md 优先 · 先保证数据 {date}-digest.md 是可 diff 的数据底座（LLM 产出最可靠）， 将来由一个渲染脚本吃 md 产出 HTML dashboard（给人看）—— 这正是仓库已有的 company.yaml → generate_concept.py → concept.svg 同款模式。 JSON 输出层暂不做，等真有"非我们自己渲染器"的机器消费者再加。
方法层已写完（M0–M4 ✅），只差"通电"。 两个数据契约（raw item / topic）、 五个采集器规格（slack 可跑；jira/github 待写但 host 有现成 MCP；gmail/feishu 已选型待接适配器）、 分诊细则、digest 模板、以及把 A→B→C 串起来的 workflows/daily_digest.yaml（时钟触发）——都已落地为文件。
唯一未通电项：M2.3 真跑一遍。 前置是 ① 外部 registry 文件（真实 channel/people ID， 在 repo 外）＋ ② MCP slack 已连。一旦备好，就能 collect→triage→render 出第一份真实 outbox/{date}-digest.md。M5（HTML dashboard）仍按计划延后。

1. 子模块拆法

秘书的工作天然是这几块，我们按职责拆开，模块间靠文件交接 （filesystem 即 message bus）——所以流程整合 (D) 放最后，没有前面几块就没东西可串。

职责本质模块
采集各 inbox（Slack / Jira / GitHub / Gmail / 飞书 …）采集——每来源独立、可并行A. collectors
合并 / 去重 / 聚类成 topic / 分级 / 打 tag分诊——合并/聚类/分级B. triage
渲染 {date}-digest.md产出C. digest md
吃 md 产 HTML dashboarddashboard 数据层E. html（暂缓 JSON）

1.3 采集器注册表（开放清单 · 能力 vs 适配器分离）

关键设计：secretary 是 host 无关的。 角色定义和 registry 只声明"能力" （"读 Slack 发我的 DM"），绝不写 host 特定的调用（如 mcp__slack__…）。 具体怎么连，是一层可插拔的适配器，由当前运行环境（host）解析：
· Claude Code → MCP server  ·  Codex → 它自己的连接器  ·  纯脚本 → REST API + token
换 host（如未来迁到 Codex）→ 换一份适配器绑定即可，角色定义和 raw item schema 一个字都不用改。

采集器不是固定的——它是插件清单，由 registry 决定启用谁。下表能力 / 采什么列是稳定契约（host 无关）； 右侧两列是同一能力在不同 host 下各自的适配器——直观展示"换 host 只换这两列、能力列不动"。 Claude Code 已验证 Codex 为前瞻规划，未验证

采集器（能力）采什么 适配器 · Claude Code 当前 适配器 · Codex 规划 可用性
slackDM / @我 / 追踪人活动 / 我的 outbound。最重，M2 先打通 MCP slackCodex connector / Slack Web API（token）✅
jiraassign 给我 / 待我转状态 / stale 超 N 天 MCP atlassianJira REST API（token）✅
github待我 review 的 PR / 被 @ 的 issue MCP githubgh CLI / GitHub REST API✅
calendar今日日程 / 会议 / 会议纪要 本地 CLI（如 gws）同 CLI（host 无关）/ Google Calendar API❌
gmail未读 / 待回邮件 Gmail MCP · 只读 scope（ArtyMcLabin fork，待接）Gmail API（OAuth token）🟡
feishu（飞书）消息 / 待办 / 文档 @ 短期 cso1z/Feishu-MCP（仅 task）→ 长期自研只读飞书 MCP（全三层）飞书开放平台 API（app token）🟡
…更多只在 owner 真正使用时新增——"grow by being used" ——⏸
读这张表的方式： 横看一行 = 同一个能力（如 slack 采"DM / @我"）， 在 Claude Code 下走 MCP，在 Codex 下走 connector / Web API——左两列永远不变，右两列随 host 替换。 这就是"能力 / 适配器分离"的具体长相。注意 calendar 这类本地 CLI 适配器其实跨 host 通用 （CLI 不依赖某个 agent runtime），所以两个 host 那格可以是同一个。
正因为采集器会很多、host 还会变，三件事是地基： ① M1 锁统一 raw item schema——无论来自 MCP / CLI / Codex 连接器，产出形状必须一致； ② 能力与适配器分离——角色规格只说"采什么"，不说"怎么调"； ③ B 绝不 hardcode 来源数——加采集器 = registry 加一项 + 写一份能力规格 + 该 host 的适配器绑定，不动 B / C。

4. 数据形状预览（M1 要锁的两个 schema）

以下是 M1 的草案方向，正式定稿时落进角色目录。先贴出来对齐认知。

raw item（A → B 契约）
一条采集到的东西，所有采集器统一产出这个形状（已定稿 → item_schema.md）
  • source 哪个采集器：slack / jira / github
  • context 定位符（#channel / repo / project-key），B 查 registry 得 group
  • type 对象是什么（给人看）：dm / mention / pr_review …
  • relation 对我意味着什么（给 B 分诊）：action_required / waiting_on_me / fyi
  • title 一句话标题
  • who 发起人（可解析 handle，便于匹配追踪人）
  • when 时间戳
  • link 回跳链接
  • raw_text 原文（供 B 聚类）
  • dedup_key 去重键（如 message ts）
topic（B → C 产出）
分诊后的一个聚合主题，digest 的最小展示单元
  • title 主题标题
  • summary 1-2 行摘要
  • priority HIGH / MED / LOW
  • group 分组（来自 registry）
  • tag 项目标签（可选）
  • participants 参与人
  • items[] 关联的 raw items

5. 不变的边界（设计约束，别漂移）

约束为什么
只读，不替我发消息/回复/建 ticket信任边界。widen 之前必须显式决定。
来源是 config，不是 prompt符合 CLAUDE.md "process is data" 铁律；真实 ID 不进公开 repo。
host 无关：只声明能力，不写 host 特定调用角色不焊死在 Claude Code/MCP 上。换到 Codex 等 host 只换适配器，角色 + schema 不变。
产出是 带日期 artifact，不是常驻服务守住仓库 "no service, no daemon"。
triage 重于 collection秘书的价值在聚类+分级，不在收集。
绝不静默丢弃 / 静默跳过Coverage 段必写："无结果" vs "没跑" 必须可区分。
A/B 是给 LLM 执行的规格（md），不是代码仓库还不是 runtime；唯一真代码是将来的 E 渲染脚本。
