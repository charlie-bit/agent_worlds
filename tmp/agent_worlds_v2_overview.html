0. 这次我们做了什么(TL;DR)

把 agent_worlds 从一个扁平的 agent 文档集,重构成一个 "一个人的 agent 公司" —— 用 公司 → 部门 → 角色 → agent 的层级组织, 并把"流程"从 prompt 里拎出来变成声明式数据(workflow YAML)。 刻意保持 V1 轻量:零运行时代码,纯文件 + 一个 workflow。

→
下一步(未做)

核心原则: 流程是数据,不是 prompt。要改公司怎么运作,改 workflow YAML; 角色之间用产物文件交接 —— 文件系统就是 message bus,无服务、无常驻进程。

2. Issue 评审:对在哪、错在哪

结论:问题感知对,解法太重、且时机错。逐条评审 —

Issue 的主张判定评审
缺的不是 Agent,而是组织 / 编排 直觉对 现有 3 个 agent 是孤岛,pm 产出 02_tasks_acceptance.md 后无下游。"缺衔接"是真问题。
流程应是系统内置能力,而非 prompt 硬编码 完全对 这是本次重构采纳的核心思想 → 落地为 workflows/software_feature.yaml。
Company→Department→Role→Workflow→Agent 层级 采纳 方向与 owner 设想一致。V1 落地,只是 Role=Agent 暂不拆(演进路径保留)。
立刻上 MessageBus(Go struct + Topic) 太重 用产物文件交接替代。仓库零后端代码,引入 Go 事件总线跨了好几个数量级。
Capability Registry 自动匹配 过早 V1 在 workflow 里直接写 role: 即可。8 个角色不需要注册中心。
DecisionCenter (Bless) 借鉴自 AgentHub 溯源错误 通读 AgentHub 的 OVERVIEW + CLAUDE.md:没有 Bless / DecisionCenter / Organization。 其 Orchestrator 只是"带 dispatch 工具的特殊 Agent",非组织架构。Issue 把自己的设想借 AgentHub 之名包装了。
6 个顶级目录 + V5 自治公司 OS 时机错 在 2/3 角色还是 TBD 空壳、连第二个完整 agent 都没有时谈摩天楼。 pm 自己写着"need 之前不要膨胀",issue 违背了项目自定的克制原则。

2.1 轻 vs 重:同一需求的两种解法

✗ Issue 的重方案(V2-V5)

  • 6 个顶级目录
  • Go MessageBus struct + Topic
  • Capability Registry 自动匹配
  • DecisionCenter 独立服务
  • 独立 runtime
  • 大量后端代码

✓ 本次的轻方案(V1)

  • 1 个 company.yaml
  • 文件系统即总线(产物文件交接)
  • workflow 里直接写 role:
  • 各角色 system/ 里一条 ADR 约定
  • 一个薄 orchestrator agent(markdown)
  • 0 行后端代码

3. 从 AgentHub 借鉴了什么(真正值得抄的)

讽刺的是,AgentHub 最该被我们抄的不是 issue 想抄的"组织运行时",而是它的文档工程 — 对我们这种文档库,零代码成本、最高回报。

AgentHub 的做法我们的采纳
四层文档分工(OVERVIEW 地图 / CLAUDE 规则 / specs 契约 / skills 配方) ✅ 采纳 新增仓库级 CLAUDE.md(规则);README 升级为 company 地图
spec 驱动 + "必须停下来问人"清单 ✅ 采纳 写进 CLAUDE.md 协作规则
单一数据源(黑名单=契约、diagram 自动生成) ✅ 强化 company.yaml 成为组织唯一事实来源 + diagram 数据源
Orchestrator = 特殊 Agent,不是独立服务 📌 借鉴 计划中的 orchestrator 也是一个 agent,非独立 runtime

4. V1 落地结构

4.1 目录(重构后)

agent_worlds/                = the company 本身
├── company.yaml             组织声明:departments → roles(单一事实来源)
├── CLAUDE.md                如何在本仓库工作(规则,新增)
├── README.md / .zh.md       公司叙事门面(双语只在顶层)
├── workflows/
│   └── software_feature.yaml  流程即数据
├── departments/             取代 assistants/builders
│   ├── product/      pm(active) · researcher(stub)
│   ├── engineering/  architect · backend · web · game(全 stub)
│   ├── quality/      qa(stub)
│   └── support/      secretary(stub)
└── diagrams/                生成的组织图(读 company.yaml)

4.2 核心 Domain 对象

company.yaml
组织的单一事实来源,也是 diagram 数据源
  • company / tagline
  • departments[] 4 个部门
  • roles[] 含 summary + status
department(部门)
按职能聚合角色
  • product 决定做什么
  • engineering 构建
  • quality 验证
  • support 承接日常
role / agent
岗位 = agent(V1 一对一)
  • agent.md 角色本体
  • system/ 方法论(可选)
  • status active / stub
workflow
把角色串成流水线;流程即数据
  • steps[] 步骤
  • consumes / produces 产物交接
  • gate owner 确认点

每步 consumes 上游产物、produces 下游产物;★ = 停下等 owner 确认

5.1 产物交接链(文件系统即 message bus)

research_notes.md      → pm 读
   ↓ pm 产出
00_project_brief.md
01_mvp_spec.md         → architect 读
02_tasks_acceptance.md → backend/web/game 读  ★scope 在此锁定(gate)
   ↓ architect 产出
architecture.md, ADR.md → build 读
   ↓ build 产出
implementation.md + code → qa 读
   ↓ qa 产出
test_report.md         → pm 读  ★发布前最终确认(gate)
一步何时能开始?当它的 consumes 文件存在;何时结束?当 produces 文件写出。 没有 daemon,没有事件总线 —— 这就是"轻"的全部秘密。
