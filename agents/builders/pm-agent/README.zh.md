# 产品规划助手 PM Agent

[English](README.md) · **中文**

把粗糙的产品想法变成小而可落地的规范包。

## 用途

帮独立创始人 / 独立开发者把粗糙的产品想法,变成一份小而可执行的规范包。

刻意保持轻量。目标是减少决策漂移,而不是制造繁重的 PM 文书。

## 角色

把 PM 决策标准化成:

- 清晰的项目简介
- 锁定的 MVP 范围
- 精简的功能规范
- 带验收标准的任务清单
- 阻塞实现的开放问题

## 何时使用

在这些时候用它:

- 一个新项目想法需要变得可落地
- 某个功能还很模糊
- 需要决定什么是 MVP、什么是以后
- 即将开始开发
- 范围在膨胀,需要砍回来

## 默认产出

每个项目都从这三个文件开始:

- `00_project_brief.md`
- `01_mvp_spec.md`
- `02_tasks_acceptance.md`

可选文件,仅在需要时添加:

- `03_open_questions.md`
- `04_change_log.md`

## 运作规则

- 优先用默认值,而不是问太多问题。
- 只问影响实现结构的问题。
- 每条需求都标 `P0`、`P1` 或 `Later`。
- 每个 `P0` 项必须有验收标准。
- 明确写出 `Out of Scope`(范围之外)。
- 不到项目需要时,不扩展成设计、开发、QA、发布等多 agent。

## 优先级标准

- `P0`:MVP 验证必需。没有它,核心循环无法被测试。
- `P1`:对 V1 质量重要,但验证 MVP 不是必需。
- `Later`:有用的未来工作,不属于当前构建。

## 决策状态

细节不明确时,用这些标签:

- `Decided`:已被 owner 确认。
- `Assumption`:为避免阻塞而用的合理默认。
- `Open`:实现前需要 owner 决策。
- `Deferred`:有意推迟。
- `Rejected`:明确不属于本版本。

## 工作流

1. 读取原始想法、笔记或 PM 决策。
2. 提取核心产品目标。
3. 定义 MVP 循环。
4. 把功能拆分为 `P0`、`P1`、`Later`。
5. 生成三文件规范包。
6. 跑 `pm-system/spec_checklist.md` 里的检查清单。
7. 只问阻塞性的开放问题。
