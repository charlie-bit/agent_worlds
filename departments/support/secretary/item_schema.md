# Secretary — Item Schema

The two data contracts the pipeline hands off on. Process is data: collectors
and triage agree on these shapes, not on each other's prompts.

```
A. collectors  ──raw item──▶  B. triage  ──topic──▶  C. digest md
```

- **raw item** (`A → B`) — one thing a collector found. Every collector, on every
  host, emits this *same* shape. This is the contract that lets B stay
  source-agnostic and lets a new source be added without touching B or C.
- **topic** (`B → C`) — one clustered subject after triage; the digest's atomic
  display unit.

Both are **specs for an LLM to follow**, not code — the repo is not a runtime
yet. Keep them stable: widening a contract is a deliberate decision, not a drift.

## raw item (A → B contract)

Every collector emits a list of these. The shape is fixed and host-agnostic —
whether the item came from an MCP server, a CLI, or a Codex connector, it looks
the same here.

| field | meaning | example |
| --- | --- | --- |
| `source` | which collector produced it | `slack` / `jira` / `github` |
| `context` | where it lives — the locator B looks up in the registry to find its `group` | `#backend-team` / `acme/widget-api` / `PROJ` |
| `type` | what the object *is* (a human-facing label) | `dm` / `mention` / `pr_review` / `ticket` |
| `relation` | what it means *for the owner* — the axis B triages on | `action_required` / `waiting_on_me` / `fyi` |
| `title` | one-line headline | `"@charlie review widget API PR"` |
| `who` | originator, as a resolvable handle (so tracked-people matching works) | `U000ALICE` / `alice` |
| `when` | timestamp (ISO 8601) | `2026-06-21T09:14:00Z` |
| `link` | jump-back URL | `https://…/archives/C000ENG/p1718…` |
| `raw_text` | the original text, for B to cluster on | `"can you take a look at…"` |
| `dedup_key` | stable identity, so B can drop duplicates without reading text | `slack:C000ENG:1718900000.123` |

### Why these fields, and not others

- **`source` vs `context`.** `source` says *which collector*; `context` says
  *where inside it*. B needs `context` to look up the registry `group`
  (channel/repo/project → group) and to render "in #backend-team". `source`
  alone can't do that.
- **`type` vs `relation` — two orthogonal axes, the key design point.**
  `type` describes the *object* (a label for humans). `relation` describes the
  *owner's obligation* and is **cross-source**: `action_required` /
  `waiting_on_me` / `fyi`. The collector — which understands its own source —
  decides `relation`; B then triages on `relation` alone and never branches on
  source. If B had to infer "needs me" from `source==github and type==pr_review`,
  it would be welded to GitHub, violating "B never hardcodes its sources." The
  digest's top sections (**Action Items** / **Waiting On Me**) map straight from
  `relation`.
- **`dedup_key` — generated by the collector, not B.** The same thing can be hit
  by more than one query layer (a boss's message both @-mentions the owner *and*
  comes from a tracked person → two identical raw items). Only the collector
  knows the source's natural unique id (Slack message `ts`, Jira issue key,
  GitHub PR/issue number), so it stamps the key; B just collapses items whose
  `dedup_key` matches — no text comparison, no source knowledge.

### `relation` values

| value | meaning | typical digest home |
| --- | --- | --- |
| `action_required` | the owner has to *do* something | Action Items |
| `waiting_on_me` | someone is blocked on the owner's reply / review / transition | Waiting On Me |
| `fyi` | context only — tracked-person activity, no obligation | Sources block |

### Deliberately *not* in the contract (add only when used)

- `thread_id` (thread clustering) — B can cluster on `raw_text` + `dedup_key`
  first; add it only if that proves insufficient.
- `status` (open/closed/done) — source-specific (Jira/GitHub); fold it into
  `raw_text` rather than letting the contract grow source-specific fields.

## topic (B → C contract)

After triage merges, dedupes, clusters, and ranks raw items, it emits topics.
One topic is one heading in the digest.

| field | meaning |
| --- | --- |
| `title` | the subject line |
| `summary` | 1–2 line summary |
| `priority` | `HIGH` / `MED` / `LOW` |
| `group` | grouping label, resolved from the registry via each item's `context` |
| `tag` | optional project tag (from registry `projects:`) |
| `participants` | people involved |
| `items[]` | the raw items this topic was clustered from |

This feeds the digest template's fixed topic line:
`- [PRIORITY] **{title}** — {summary} ({participants})`.

## Adding a source (what stays fixed)

When a new collector is added: it must emit the **raw item** shape above —
including a source-correct `dedup_key` and a `relation` it judged itself. B and C
do not change. That invariant is the whole point of pinning these two contracts.
